# 테크니컬 라이팅
- 기술적인 내용을 효과적으로 전달하는 글쓰기법
- 중요한 점은 독자가 문제를 해결하고 목표를 이루도록 도울 수 있어야 한다
- 정보를 빠르고 정확하게 전달하는데 집중하자

## 좋은 기술 문서를 쓰는 핵심원칙
- 좋은 기술 문서를 작성하려면 
	- 문서의 유형을 파악할 수 있어야 한다
	- 정보를 효과적으로 구조화 할 수 있어야 한다
	- 이해하기 좋은 문장을 쓸 수 있어야 한다

### 문서의 유형
- 먼저 문서를 읽는 목적을 설정해야 한다
- 목적에 맞는 문서 유형은 이미 어느 정도 정해진 구조가 있다
- 그 구조를 이해하고 작성해야한다

- API 레퍼런스?, 튜토리얼?, FAQ?
- 각각에 맞는 효과적인 구조로 내 머릿속에 있는 지식을 재구성하자

#### 대표적인 문서 유형
1. 학습 중심 문서
	- 새로운 기술이나 도구에 대한 흐름을 다루는 문서
	- 튜토리얼 문서, 사전 준비 문서다
2. 문제 해결 문서
	- 배경 지식이 있는 상태에서 기술의 특정 문제를 해결하려는 문서
3. 참조 문서
	- 기본적인 작동 방법을 아는 상태에서 특정 기능, API 사용법을 확인해 적용하고 싶을 때 쓰는 문서
	- 레퍼런스 문서다
4. 설명 문서
	- 개념, 원리, 배경 지식의 깊은 이해를 다루는 문서

#### 각 유형별 샘플
- 학습 중심
```md
# [제목]

## 목표
- 이 튜토리얼을 따라하면 독자가 할 수 있는 일이 무엇인가?
  
## 사전 요구사항
- 튜토리얼 진행에 필요한 지식, 도구 정리 등
  
## 단계별 가이드

### [첫번째 단계 제목]
- 작업 설명 & 코드 예제
### [두번째 단계 제목]
- 반복
...

### 최종 결과 확인
[목표 달성 시 어떤 결과가 나오는지 설명] 
```

- 문제 해결 중심
```md
# [제목]

## 문제 정의
- 문제 상황을 설명
  
## 사전 요구사항
- 환경 설정, 필수 조건 등
  
## 해결 방법

### [첫번째 해결 방법]
- 솔루션 첫 번째

### [두번째 해결 방법]
- 솔루션 두 번째
...  

### 문제 해결 후 확인 방법
[해결 후의 모습, 확인 하는 법 설명]
```

- 참조 문서
```md
# [제목]

## 개요
[이 요소가 무엇인지, 언제 사용하는지, 어떤 가치를 제공하는지]

## 속성 및 옵션
|속성명|타입|기본값|설명|
|---|---|---|---|
...

## 시그니처
[예제 코드]

## 반환 값
[함수나 API 가 반환하는 값]

## 사용 예제
[어떤 상황에서 사용하는지 구체적 예시, 예제 코드]
```

- 설명 문서
```md
# [개념명]

## 개념 소개
[간략한 개념 소개]

## 등장 배경
[이 개념이 필요해진 등장 배경]

## 활용
[실제 프로젝트에 활용되는 개념 예시]
```

### 정보 구조
- 정보가 잘 구조화되어 있다면 새로운 지식을 더 쉽게 이해할 수 있다
- 문서를 읽는 입장은 많이 해봤겠지만 쓰는 입장은 또 다르다
- 문서를 쓰는 입장에서 읽는 사람 입장을 생각해야한다
- 내가 말하고 싶은 내용 위주로 구조화하면 안된다

#### 주요 원칙
- 한 페이지에 하나만 다루기
	- 제목 깊이가 `####` 이상 이면 문서 나누기를 고려하기
	- 개요를 다루는 페이지를 활용해보자
	- 좀 더 구체적인 주제를 다루도록 해보자
- 가치를 먼저 제공하기
	- 독자 입장에서 얻을 가치를 먼저 전달하기
	- 부가적 정보나 세부 사항은 나중으로
	- 기능 중심의 설명을 피해라
		- 이 스니펫은 다양한 설정 옵션을 제공합니다. 먼저 `connection_timeout`, `retry_attempts`, `max_pool_size` 등의 매개변수를 수정하세요. 그런 다음, ... (x)
		- 이 스니펫을 사용하면 PostgreSQL 데이터베이스 연결 속도가 50% 빨라집니다 (o)
		- 핵심 가치를 먼저 제시하라
- 효과적인 제목 쓰기
	- 핵심 키워드 포함하기
	- 일관성 유지
	- 30자 이내 간결
	- 평서문
- 개요 빠트리지 않기
	- 개요에 문서의 핵심 내용을 요약
	- 문서의 목표를 독자에게 정확히 전달
	- 기술적 배경 설명 보다 핵심 정보를 먼저 제공
		- 이 문서는 TypeScript의 유틸리티 타입을 소개합니다. Partial, Pick, Omit 등의 유틸리티 타입을 사용할 수 있어요 (x) 왜 유틸리티 타입을 사용해야 하는지 설명이 부족
		- TypeScript의 유틸리티 타입을 사용해서 객체 타입을 변형하는 방법을 알아볼게요. 유틸리티 타입을 활용하면 반복적인 타입 선언을 줄이고, 더 유연하게 타입을 관리할 수 있어요. 이 문서에서는 Partial, Pick, Omit 등을 사용하여 객체 타입을 효과적으로 다루는 방법을 알아볼게요. (o) 핵심 정보, 목표 먼저 제공
- 예측 가능하게 하기
	- 일관된 제목, 형식, 구조가 글을 예측 가능하게 한다
	- 섹션 제목을 동일한 패턴으로 맞춰라
	- 논리적 순서로 정보 배치
	- 용어를 일관되게 하라

### 문장 다듬기
- 기술 문서는 명확하고 이해하기 쉬워야 한다

### 주요 원칙
- 문장의 주체를 분명하게 하기
	- 개발자가 주체임을 표현하라
		- '이 라이브러리는' 데이터베이스를 .. (x)
		- 이 명령어를 실행하면 데이터베이스를 .. (o)
		- 물론 예외도 있을 수 있다
	- 능동형으로 표현하라
		- 변경 사항이 적용된 후 다시 빌드되어야 합니다 (x)
		- 변경 사항을 적용한 후 다시 빌드하세요 (o)
		- 애플리케이션이 실행되기 전에 설정이 완료되어야 합니다 (x)
		- 애플리케이션을 실행하기 전에 설정을 완료하세요 (o)
- 필요한 정보만 남기기
	- 문장을 짧고 간결하게 유지하기
		- 버튼을 클릭하면 다음 단계로 이동하게 되며, 그 이후의 작업을 진행할 수 있습니다 (x)
		- 버튼을 클릭하면 다음 단계로 이동합니다. 이동 후 이어서 작업을 진행하세요 (o)
		- 이 API를 호출할 때 요청 헤더를 포함해야 하며, 올바른 인증 정보를 제공해야 정상적으로 응답을 받을 수 있습니다 (x)
		- 이 API를 호출할 때 요청 헤더와 인증 정보를 포함하세요 (o)
	- 메타 담화를 최소화 하기
	- 메타 담화란 핵심 메시지 자체가 아니라 그 메시지를 보조하는 표현을 말한다
		- 앞에서 설명했지만, 결론은 이 설정을 변경하는 것이 가장 효과적인 방법이라는 겁니다 (x)
		- 이 설정을 변경하는 것이 가장 효과적인 방법입니다 (o)
		- 아시다시피, 이 설정은 성능에 영향을 줍니다 (x)
		- 이 설정은 성능에 영향을 줍니다 (o)
- 구체적으로 쓰기
	- 명사 대신 동사 쓰기
	- 불필요한 명사가 포함되거나 의미가 함축된 명사를 경계해야 한다
		- 코드 최적화 진행 후 배포 수행이 필요합니다 (x)
		- 코드를 최적화한 후 배포하세요 (o)
		- MongoDB 연결 정보 설정 및 초기화가 필요합니다 (x)
		- MongoDB에 연결할 호스트와 포트를 설정하고 데이터베이스를 초기화합니다 (o)
	- 모호한 표현 대신 명확한 표현을 사용하기
		- 설정 파일을 변경하면 기존 설정이 영향을 받을 수도 있습니다 (x)
		- 설정 파일을 변경하면 기존 설정이 삭제됩니다 (o)
		- 일부 브라우저에서 정상적으로 동작하지 않을 가능성이 있습니다 (x)
		- IE 에서는 정상적으로 동작하지 않습니다 (o)
- 자연스러운 한국어 표현 쓰기
	- 불필요한 한자어 제거
		- 로그 파일을 삭제하는 작업을 수행합니다 (x)
		- 로그 파일을 삭제합니다 (o)
	- 번역체 표현 수정하기
		- API 키를 이용한 사용자 인증 처리가 완료된 후, 데이터베이스 접속 설정 진행이 가능합니다 (x)
		- API 키로 사용자를 인증한 후, 데이터베이스에 접속하도록 설정할 수 있습니다 (o)
		- 이 API를 통해 데이터를 가져올 수 있습니다 (x)
		- 이 API로 데이터를 가져올 수 있습니다 (o)
- 일관되게 쓰기
	- 공식 기술 용어 따르기
	- 같은 개념을 여러 방식으로 표현하지 말기
		- 파일을 추가하려면 '파일 선택' 버튼을 클릭하세요. 파일을 첨부한 후 '저장'을 누르면 업로드가 완료됩니다. 필요한 경우 파일을 다시 넣을 수 있습니다 (x)
		- 파일을 업로드하려면 '파일 선택' 버튼을 클릭하세요. 파일을 업로드한 후 '저장'을 누르면 업로드가 완료됩니다. 필요한 경우 파일을 다시 업로드할 수 있습니다 (o)
	- 약어는 먼저 풀어쓴 후 사용하기
		- 이 기능은 SSR을 지원합니다 (x)
		- 이 기능은 SSR(Server-Side Rendering) 을 지원합니다 (o) 이후부터 약어 사용
	- 외래어 표기는 가급적 적게 쓰기
	- 외래어 표기는 맞춤법보다는 실제로 통용되는 일반적 표현을 사용하기
		- 프런트엔드 (x)
		- 프론트엔드 (o) 구글 트렌드를 기준으로 잡으면 좋다, 실제로 5배 이상 차이남
