# Parallel Stream 원리와 성능 최적화

# 병렬화는 항상 옳은가?
Stream API 의 강력한 특징 중 하나는 `parallel()` 메서드 호출로 손쉽게 병렬 처리를 지시할 수 있다는 점이다.

```java
// 순차 처리 Stream
long count = list.stream().filter(i -> i > 10).count();

// 병렬 처리 Stream
long count = list.parallelStream().filter(i -> i > 10).count();
```

하지만 병렬 처리가 무조건 순차 처리보다 우월한건 아니다.
내부 동작 원리를 이해하고 어떤 때에 병렬 처리가 유리한지 알아보자.

# Spliterator
`Iterator` 인터페이스는 순차적 순회로 구현되어 있어 데이터를 나누는 작업을 맡길 수가 없다. 이를 개선한게 `Spliterator` 인터페이스 이다.

## 핵심 메서드
- `tryAdvance(Consumer action)`: 순차 처리를 담당하는 메서드, 요소를 하나 가져와서 `Consumer` 인 `Sink` 에게 전달하고 다음 요소가 있으면 `true` 없으면 `false` 를 반환한다

- `trySplit()`: 병렬 처리를 담당하는 메서드, 데이터 일부를(보통 절반) 떼어내서 새로운 `Spliterator` 객체를 만들어 반환하고 자신은 나머지(절반)만 관리한다. 더 이상 분할할 수 없으면 `null` 을 반환한다.

# 자료구조에 따른 분할 성능
병렬 스트림이 호출되면 `Fork/Join Framework` 가 `trySplit()` 을 재귀적으로 호출한다. 이때 대상이 되는 자료구조에 따라 분할 비용이 달라진다.

## 인덱싱 기반, Random Access 자료 구조의 경우

### ArrayList, 배열, IntStream.range 등
- **최상의 효율**
	- 메모리에 연속할당 된 자료구조로 `mid = (start + end) / 2`  를 통해 중간점을 바로 구한다
	- O(1), 데이터 크기와 상관없이 인덱스로 바로 절반으로 쪼개진다
	- 병렬 처리 시 스레드 간의 작업 분할이 완벽하게 균등해진다

### HashSet, HashMap
- **좋음**
	- 내부에 버킷이라는 배열을 가지는 자료구조, 버킷 안에 노드들이 구현되는 형태
	- 버킷 배열을 절반으로 나누는 작업은 배열이라 O(1) 이지만, 버킷 간의 데이터 편차가 있을 수 있어 ArrayList 만큼 균등하진 않다
	- 버킷마다 데이터가 많을 수도, 적을 수도 있기 때문에 ArrayList 만큼 완벽하지는 않지만 꽤 효율적인 병렬처리가 이루어진다

## 노드 기반, Sequential Access 순차 접근 자료 구조의 경우

### LinkedList
- **최악 효율**
	- 메모리에 노드들이 흩어져 있고 주소로 연결된 자료구조이다
	- 분할 지점을 찾기 위해 `N/2` 번 이동해야 해서 O(N) 으로 큰 비용이 든다
	- 균등 분할이 어려워 스레드 간 작업 부하 불균형이 발생한다

### TreeSet, TreeMap
- **좋음**
	- 노드가 왼쪽, 오른쪽으로 구분되기 때문에 분할이 쉽다
	- 분할 방식이 비교적 저렴하고 트리가 완전히 균등함을 보장하진 않지만 어느정도 보장된다

## 효율표
| **효율** | **자료구조**                                | **분할 비용** | **데이터 균등성** | **추천 여부**  |
| ------ | --------------------------------------- | --------- | ----------- | ---------- |
| **최고** | `ArrayList`, `Array`, `IntStream.range` | **O(1)**  | **Perfect** | **강력 추천**  |
| **좋음** | `HashSet`, `HashMap`                    | O(1)      | Good        | 추천         |
| **좋음** | `TreeSet`, `TreeMap`                    | Low       | Very Good   | 추천         |
| **나쁨** | `LinkedList`, `IO Stream`               | **O(N)**  | **Bad**     | **절대 비추천** |

# Fork/Join 프레임워크
병렬 스트림은 단순히 스레드 여러 개를 생성하는 것이 아니라 Java 7 부터 도입된 Fork/Join 프레임워크 를 기반으로 동작한다.

## 분할 정복
거대한 작업을 작은 단위로 쪼개고(Fork) 각 스레드가 병렬로 처리한 뒤 결과를 다시 합치는(Join) 과정을 말한다.

1. `Fork`: `Spliterator` 가 데이터를 더 이상 나눌 수 없을 때까지 분할한다
2. `Process`: 각 스레드는 할당받은 데이터를 처리한다
3. `Join`: 각각 처리된 결과를 병합해 최종 결과를 도출한다

## 공용 풀, 작업 훔치기 Common Pool, Work-Stealing

### 공용 풀
병렬 스트림은 별도의 설정이 없다면 JVM 전역에 단 하나 존재하는 `ForkJoinPool.commonPool()` 을 공유한다. 스레드 개수는 기본적으로 `CPU 코어 수 - 1` 로 설정된다.

즉, 하나의 병렬 스트림이 네트워크 I/O 등으로 인해 블로킹 되면 애플리케이션 전체의 병렬 스트림이 멈추는 대형 장애로 이어질 수 있다

### 작업 훔치기
작업 훔지기(Work-Stealing) 알고리즘은 병렬 스트림 효율의 핵심이다.
- 모든 스레드는 각자의 `Deque` 작업 큐에 작업을 쌓아두고 처리한다
- 만약 자신의 할 일을 다 끝냈다면 **바쁜 스레드의 덱 뒤 쪽에 있는 작업을 훔쳐와 처리한다**
- 이를 통해 CPU 자원을 유휴 상태 없이 최대한 효율적으로 사용한다

# 병렬화 성능
핵심은 '작업을 병렬화 하는 작업'도 비용이 든다는 점이다.

```
총 처리 시간 = 
분할하는데 드는 비용 + 
실제 작업 처리 시간 + 
나눠진 결과물 병합 비용 + 
스레드 간 컨텍스트 스위칭 및 캐시 미스 비용
```

실제 작업 처리 시간은 병렬화에 의해 N분으로 나누어 진다.
즉, 나머지 비용을 다 합쳐서 '싱글 스레드 실제 작업 처리 시간' 보다 짧아야 병렬화가 의미를 가진다.

## NQ 모델
Java 설계자 Brian Goetz 가 제시한 모델로 병렬화 처리 여부를 결정할 수 있는 가이드이다.

```
N * Q > 10,000

N: 데이터 요소의 개수
Q: 요소 하나를 처리하는 데 걸리는 비용
```

- N 이 클 때: 데이터가 1만 개 이상으로 매우 많다면, Q 가 낮아도 병렬화 효과가 유리하다
- Q 가 클 때: 데이터가 적어도, 요소 하나를 처리하는 비용(복잡한 암호화 연산 등)이 매우 비싸다면 병렬화가 유리하다
- 둘 다 작다면
	- N 이 1,000 개 이하로 매우 작다면 병렬화 오버헤드가 더 커져버리기 때문에 순차 스트림을 써야 한다
	- Q 가 매우 작다면(`int` 덧셈 연산 등) N이 매우 커야 병렬화에 의미가 생긴다

## 병합 비용
쪼개진 작업을 다시 합치는 비용도 중요하다.

### 종단 연산
`sum()`, `max()`, `min()`, `count()` 등은 숫자끼리 더하거나 비교하는 작업으로 가격이 매우 싸기 때문에 병렬화에 유리하다.

`collect(Collectors.toSet())`, `Collectors.toMap()` 등은 병합 비용이 비싸다. 서로 다른 두 개의 `set` 이나 `map` 을 하나로 합치는 과정은 내부적으로 엄청난 데이터 복사와 재해싱이 필요해 병렬화의 이점이 상쇄될 수 있다.

### 박싱/언박싱

```java
// 비효율적인 코드, Integer 객체 생성 비용 발생
Stream.iterate(1, i -> i + 1).limit(10000).parallel().reduce(0, Integer::sum);
```

일반 스트림 `Stream<Integer>` 은 기본형 `int` 를 래퍼 클래스 `Integer` 로 감싸는 박싱 비용이 발생한다.
성능을 위해서 반드시 기본형 스트림(`IntStream`, `LongStream`, `DoubleStream`) 을 사용해야 한다.
